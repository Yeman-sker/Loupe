# ADP: Local-project-gated onboarding and daemon status surfaces

## Context

Loupe is primarily a UI-improvement tool for local development projects. Two current
surface behaviours do not match that product boundary:

1. The current unauthorized-host card separates the explanatory prompt from the real
   Chrome host-permission CTA. It tells the user to click the browser toolbar icon rather
   than presenting the prototype's direct **Allow site** action.
2. The extension currently treats any new http/https origin as a candidate Loupe surface.
   That means ordinary websites can show the Loupe prompt even when they are not local
   development projects.

There is also a daemon auth/onboarding question. The PRD requires loopback APIs
(`/v1/marks*`, `/mcp`) to use a Bearer token stored at `~/.loupe/token`, but exposing a
manual token input to users would make the product feel like infrastructure setup rather
than a local UI workflow.

This intersects with `docs/adp-20260605-host-auth-grant-via-toolbar.md`, which accepted a
toolbar-only permission grant because MV3 cannot preserve the user gesture when an
in-page content-script click is relayed to a service worker.

## Decision

Loupe onboarding should become **local-project gated** and **status driven**:

1. **Do not add a normal daemon token input UI.** The token remains a local trust artifact
   generated by `loupe init` and read by the daemon / CLI / MCP proxy / extension
   background pairing path. Page surfaces must not display or request the token value.

2. **Represent daemon auth as a status surface, not a form.** The UI may show states like
   connected, daemon offline, token missing/invalid, sync failed, local-only saved, retry,
   copy Markdown, and `loupe init` repair guidance. Advanced diagnostics may show
   `token_path` such as `~/.loupe/token`, but not the token value.

3. **Only show Loupe host-authorization onboarding for local project candidates.** Before
   rendering the host-auth prompt, the extension should classify the current origin as a
   local development project candidate. Obvious candidates include loopback hosts
   (`localhost`, `127.0.0.1`, `::1`), private-network dev hosts, and origins associated
   with a known Loupe workspace/project identity. Ordinary public websites should stay
   silent unless the user explicitly invokes Loupe there.

4. **Restore a prototype-faithful host-auth CTA.** The user-facing unauthorized-host
   surface should present an **Allow site** primary button, matching the prototype.
   Implementation must still respect MV3 gesture constraints: if an in-page button cannot
   directly call `chrome.permissions.request`, the product should use an MV3-valid
   equivalent such as an action popup with the same CTA semantics, or another browser
   action path that preserves a real user gesture. The final UX should not make the user
   infer that the toolbar icon is the primary CTA.

## Alternatives considered

- **Expose a token input in the in-page UI.** Rejected: worse UX, higher leakage risk near
  the host page boundary, and unnecessary because local tooling can read `~/.loupe/token`.
- **Keep showing Loupe on every untrusted new origin.** Rejected: noisy for a tool whose
  core use case is local UI development, and likely to reduce user trust.
- **Keep the current informational CTA that points at the toolbar.** Rejected as the target
  UX: it is technically safe but visually diverges from the locked prototype and adds
  cognitive overhead.
- **Literal in-page `chrome.permissions.request`.** Rejected as an implementation detail
  unless proven viable in the active MV3 context; content-script clicks can lose the
  transient user gesture. The target is prototype-faithful CTA semantics, not necessarily
  a literal content-script API call.

## Consequences

- This decision adds a discovery gate before Surface 1: Loupe should decide whether the
  origin looks like a local project before showing host-auth UI.
- Generic public websites remain quiet by default. Users can still explicitly invoke
  Loupe later if the product chooses to support non-local origins.
- The daemon/pairing surface becomes a compact status/diagnostic strip rather than a token
  form.
- If accepted, this supersedes the user-facing part of
  `docs/adp-20260605-host-auth-grant-via-toolbar.md`; the toolbar grant path can remain as
  a fallback implementation, but not as the primary visible CTA.
